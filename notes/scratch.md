# Performance Examples Overview

This directory contains performance experiments demonstrating various Go optimizations and trade-offs.

## bm - Basic Benchmarks

### reserve - Slice Allocation Strategy
Demonstrates the performance impact of pre-allocating slices vs appending.

- **Unpreallocated** (`var vs [][]int`): 15 allocations, ~11,000 ns/op
- **Preallocated** (`make([][]int, 10)`): 11 allocations, ~8,000 ns/op

Pre-allocating the outer slice reduces allocations by avoiding dynamic resizing.

### find - String Search Methods
Compares different approaches to find a character in a string:

| Method | Small string (21 chars) | Large string (163KB) |
|--------|------------------------|---------------------|
| `strings.Contains` | 12.89 ns/op | 3,784 ns/op |
| Manual iteration | 19.42 ns/op | 126,495 ns/op |

`strings.Contains` uses optimized assembly (likely SWAR) for large strings, making it faster for large inputs despite being slower for tiny strings.

## extra - Map vs Slice Access

Compares access patterns:

- `MapAccess`: Standard Go map lookup
- `SliceAccess`: Direct slice indexing
- `FNV`: FNV-1 hash computation
- `Quasi1/Quasi2`: Custom hash-based slice indexing

Maps are slower than slices for direct access but provide flexibility with arbitrary keys.

## findchar - Finding Semicolon

Compares methods to find a semicolon separator in text data:

- `strings.Cut`: String method
- `strings.Split`: Full split (allocates)
- `strings.Index`: Index lookup
- Manual iteration: Character-by-character scan
- **SWAR**: Uses SIMD-like bit manipulation to check 8 bytes at once

SWAR is fastest for longer strings, but has overhead that makes it slower for very short strings.

## getpagesize - System Page Size

Shows how to query the OS page size via `os.Getpagesize()`. Important for understanding memory alignment and mapping boundaries.

## mapkey - Key Conversion Methods

Compresents various ways to convert byte slices to map keys:

- `ByteSliceToString`: Standard conversion (allocates)
- `ByteSliceToStringZeroAlloc`: Unsafe pointer cast (no allocation)
- `ByteSliceToFNV`: FNV-1a hash
- `ByteSliceToIntKey`: Simple hash function
- `ByteSliceToBigInt`: Big integer conversion
- `Djb2`: Daniel Bernstein's hash
- `FxHasher`: Firefox's fast hasher using rotation

For key comparison performance, custom simple hash functions often outperform standard library options.

## parsetemp - Temperature Parsing

Compares parsing temperature strings like "17.3":

| Method | Performance |
|--------|-------------|
| `strconv.ParseFloat` | 20.68 ns/op |
| `big.ParseFloat` | 482.6 ns/op |
| `ParseTempToInt` | 4.01 ns/op |
| `ParseNumber` (custom) | 0.25 ns/op |

Custom parsing for known formats is dramatically faster than general-purpose parseFloat. The simple byte-by-byte parser avoids strconv overhead.

## read - File Reading Methods

Compares file I/O performance with different buffer sizes:

| Method | Pattern |
|--------|---------|
| `Raw` | Unbuffered `Read()` calls |
| `ReadByte` | Buffered, single byte |
| `ReadString` | Buffered, line-to-string |
| `ReadBytes` | Buffered, line-to-byte-slice |

Larger buffers reduce syscall overhead. Unbuffered I/O is slowest; buffered methods scale better with buffer size (optimal around 64KB-256KB for this workload).

## readms - Memory Statistics

Demonstrates `runtime.ReadMemStats` to capture heap allocations and memory usage. Useful for understanding the allocation behavior of other benchmarks.

## scanbuf - Scanner Buffer Size

Tests how scanner buffer size affects throughput when reading large files:

- Small buffer (1KB): ~14,200 ms
- Medium buffer (64KB): ~8,800 ms
- Large buffer (16MB+): ~8,700-9,700 ms

Optimal buffer size is typically between 64KB-256KB; beyond that, benefits plateau.

## semi - SWAR Semicolon Detection

Implements SWAR (SIMD Within A Register) for finding semicolons:

1. Load 8 bytes into uint64
2. XOR with broadcast separator value
3. Use bit manipulation to detect matches
4. Use `bits.TrailingZeros64` to find position

This processes 8 bytes per operation vs 1 byte for naive iteration.

## split - String Splitting

Compares splitting methods:

| Method | Short string (9 chars) | Long string (31 chars) |
|--------|----------------------|----------------------|
| `strings.Split` | 65.26 ns/op | 61.59 ns/op |
| `bytes.Split` | 91.13 ns/op | 59.67 ns/op |
| **Custom** | **3.60 ns/op** | **7.99 ns/op** |

Custom split (hand-written loop) is 8-18x faster than standard library because it avoids allocation and handles the specific use case (splitting on known separator).

## swar - SWAR Operations

Demonstrates SIMD-within-a-register techniques:

- `IsAsciiSlow`: Byte-by-byte high-bit check
- `IsAsciiSwar`: Check 8 bytes at once using bitmask
- `AddUint16Swar`: Add 4 uint16 values in parallel using pseudo-lanes

SWAR gives 4-8x speedup for bulk operations on data aligned to word boundaries.

## xmap - Custom Map Implementations

### custom - Hash Table with Open Addressing

Custom map using:
- FNV-64 hash
- Linear probing for collision resolution
- Pre-allocated bucket array

Faster than standard map for known-size workloads with predictable access patterns.

### fixed - Static Map for City Data

Tailored map for city temperature data:
- Pre-allocated array of 16,384 slots
- Custom hash function optimized for city names
- No map overhead, direct array indexing

Specialized structures beat general-purpose maps when data pattern is known ahead of time.
